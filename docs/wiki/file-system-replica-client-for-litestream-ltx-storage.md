## Overview

The file replica client is the simplest backend in the litestream replication system. It maps every operation in the `ReplicaClient` interface—listing, opening, writing, and deleting LTX files—directly to filesystem calls under a configured base path. Despite its simplicity, it is a fully first-class backend that supports both the v2 (level-based LTX) and v3 (generation-based WAL segment) replication protocols.

## Registration and URL Parsing

At package init time the client registers itself with litestream's global client factory under the scheme `"file"`. URLs of the form `file:///var/replicas/mydb` are parsed by `NewReplicaClientFromURL`, which validates that the URL contains a non-empty path and delegates to `NewReplicaClient(path)`.

## Directory Layout

Files are organized using helpers from the core litestream package:

```
<base>/ltx/<level>/<minTXID>-<maxTXID>.ltx
```

`LTXLevelDir` and `LTXFilePath` convert level integers and transaction IDs into this structure. The path helpers call `filepath.FromSlash` so that forward-slash paths generated by the shared helper work correctly on Windows.

## Listing LTX Files

`LTXFiles` opens the level directory and calls `Readdir` in a single shot rather than streaming entries. This is intentional for a local filesystem—random access is cheap, and the full listing fits comfortably in memory. Non-LTX filenames are skipped by attempting to parse each filename with `ltx.ParseFilename`; parse failures are silently ignored. The `useMetadata` parameter accepted by the interface is ignored for the file backend because `ModTime` is always accurate and always available from the directory entry—there is no network round-trip to save.

## Writing with Timestamp Preservation

`WriteLTXFile` is the most complex method. The critical design choice is that it uses `io.TeeReader` to peek at the LTX header while simultaneously buffering that data so the complete byte stream can be written to disk afterward. The header contains a millisecond-resolution `Timestamp` field, which is extracted and then applied to the final file via `os.Chtimes`. This matters for restore operations: the LTX file's `CreatedAt` metadata (used by the iterator) must reflect the original transaction time, not the time of the replication copy. Without this step, timestamp-based point-in-time restore would use wall-clock time of the backup run rather than the time the transaction was committed.

The write itself uses a temp file pattern—write to a `.tmp` file first, then rename atomically—to avoid leaving partial files visible to concurrent readers during an incomplete write.

## Partial Reads via Offset and Size

`OpenLTXFile` supports seeking to an arbitrary byte offset within a file and capping reads with an `internal.LimitReadCloser`. This is used by the resumable reader logic at higher layers, which can reopen a dropped stream from the exact byte position where it stopped.

## v3 Protocol Methods

The client also implements `ReplicaClientV3`, exposing `GenerationsV3`, `SnapshotsV3`, `WALSegmentsV3`, `OpenSnapshotV3`, and `OpenWALSegmentV3`. These map to corresponding subdirectories under the base path for generation-UUID-keyed storage used by the newer restore path.

## Known Gaps

None explicitly flagged in the source. The `Init` method is a deliberate no-op—file backends require no connection establishment—and this is documented inline.
