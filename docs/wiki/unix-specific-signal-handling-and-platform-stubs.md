## Overview

Litestream supports running as a Windows Service in addition to a regular Unix daemon. Rather than scattering `runtime.GOOS` checks throughout the main logic, the codebase splits the three platform-sensitive functions into two files selected by build tags:

- `main_notwindows.go` (this file) — compiled on Linux, macOS, and all non-Windows targets
- `main_windows.go` — compiled only on Windows

This keeps `main.go` clean and ensures the compiler enforces that both files provide the same function signatures.

## Functions

### `isWindowsService() (bool, error)`

Always returns `(false, nil)`. On Unix there is no concept of a Windows Service Control Manager, so the function is a no-op stub. The caller in `main.go` uses the return value to decide whether to redirect logging to the Windows Event Log and hand control to the SCM; on Unix this path is never taken.

### `runWindowsService(ctx context.Context) error`

Panics unconditionally with `"cannot run windows service as unix process"`. This function is called only when `isWindowsService()` returns true, which never happens on Unix. The panic rather than a graceful error is intentional: if this code path were somehow reached, it would indicate a programming error rather than a user error. A panic produces a clear stack trace that surfaces the bug immediately.

### `signalChan() <-chan os.Signal`

Creates a buffered channel of size 2 and registers it to receive `SIGINT` and `SIGTERM`. The buffer size of 2 matters: if the main goroutine is busy processing the first signal when a second arrives (for example, a user pressing Ctrl+C twice), the second signal is not dropped. Without buffering, Go's `signal.Notify` would silently discard signals that arrive before the channel is drained.

`SIGTERM` is the standard signal sent by init systems, Docker, and Kubernetes for graceful shutdown. `SIGINT` is generated by Ctrl+C. Registering both ensures litestream shuts down cleanly whether stopped by an orchestrator or by a developer.

## Default Config Path

The file also sets `defaultConfigPath = "/etc/litestream.yml"`, the conventional Unix location for system-wide configuration files. The Windows counterpart sets a `C:\Litestream\litestream.yml` path.
